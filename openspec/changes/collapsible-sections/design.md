## Context

Fusen の Live Preview エディタは `MarkdownStyler` が swift-markdown の AST を解析し、`NSAttributedString` に変換することでレンダリングを実現している。`BulletLayoutManager` がカスタム描画を担い、箇条書きや blockquote の border を描画している。折りたたみ機能は、この既存レンダリングパイプラインに折りたたみ状態の概念を追加する形で実装する。

現在のアーキテクチャ:
- `LivePreviewEditor` (NSViewRepresentable) — NSTextView をラップ
- `MarkdownStyler` — AST → NSAttributedString 変換、カーソルブロック判定
- `BulletLayoutManager` — カスタム NSLayoutManager

## Goals / Non-Goals

**Goals:**
- セクション（見出し）・リスト・TODOリストを折りたたみ/展開できる
- 折りたたみ状態をノートごとに永続化する（state.yaml）
- カーソルが折りたたみ可能なブロック上にあるときトグルボタンを表示する
- 折りたたんだ箇所の左に `>` インジケーターを表示する

**Non-Goals:**
- 任意のブロック（blockquote、コードブロック等）の折りたたみ
- キーボードショートカットでの折りたたみ操作（将来の拡張）
- 折りたたみ状態のエクスポートや他ツールとの共有

## Decisions

### 折りたたみ状態の管理: インメモリ + state.yaml 永続化

**選択**: `FoldingState` 構造体をノートごとに保持し、折りたたまれた範囲をソース行番号または見出し/リスト識別子のセットで管理する。state.yaml に保存する。キーは**解決済みファイルパス**（テンプレートではなく実際のパス）とする。

**代替案**:
- ファイル内にメタデータを埋め込む → ファイル内容を変更することになり Obsidian 互換を損なうため却下
- NSTextStorage の attribute として管理 → 永続化が難しいため却下

**理由**: state.yaml はすでにウィンドウ位置等の揮発的状態を管理しており、折りたたみ状態もその延長で管理するのが自然。解決済みパスをキーにすることで、Periodic Note（日付テンプレートで日ごとにパスが変わるノート）は自然と日ごとに独立した折りたたみ状態を持つ。翌日の新しいパスでは折りたたみ状態なしのクリーンな状態から始まる。

### 折りたたみ対象の識別: 見出しレベル + 開始行番号

**選択**: 折りたたみ単位を「見出し（H1〜H6）に続くコンテンツ」と「リスト全体（トップレベルのリストアイテム）」とする。識別子はソース内の開始行番号で管理する。

**代替案**:
- UUID を振って管理 → ファイル変更時に UUID が無効になる問題
- テキスト内容ハッシュ → ファイル更新時に衝突しやすい

**理由**: 行番号はファイル変更で変わりうるが、折りたたみ状態は近似的で良く、行番号ズレは許容範囲。

### レンダリング: `MarkdownStyler` で折りたたみ範囲をスキップ

**選択**: `MarkdownStyler.style()` 呼び出し時に折りたたみ状態を受け取り、折りたたまれたブロックの子ノードを `NSAttributedString` に追加しない（非表示範囲を生成しない）。

**代替案**:
- `NSTextView` の `isHidden` 制御 → NSTextView は範囲単位での非表示が困難
- NSLayoutManager での行高ゼロ化 → 複雑で副作用が多い

**理由**: AST 段階でスキップするのが最もシンプルで確実。`MarkdownStyler` はすでに AST を走査しているため、折りたたみ判定を追加しやすい。

### トグルボタン: NSTextView 上の NSButton オーバーレイ

**選択**: カーソルがある折りたたみ可能なブロックの先頭行に対して、`NSTextView` 上にオーバーレイで `NSButton`（シェブロン画像）を配置する。カーソル移動時に動的に表示/非表示を切り替える。

**代替案**:
- `NSTextAttachment` として埋め込む → テキストの一部になりコピー時に混入する
- BulletLayoutManager で描画 → クリックイベントが取得できない

**理由**: オーバーレイ NSButton はクリックイベントが自然に取れ、表示位置計算も `layoutManager.boundingRect(forGlyphRange:...)` で対応可能。

### 折りたたみインジケーター `>`: BulletLayoutManager で描画

**選択**: `BulletLayoutManager` の `drawBackground` / `drawGlyphs` で、折りたたまれたブロックの「最後の表示行」の左マージンに `>` を描画する。

**理由**: 既存のカスタム描画と統一できる。

## Risks / Trade-offs

- **行番号による識別の脆弱性**: ファイルが外部から変更されると折りたたみ状態がズレる → 起動時にファイルを再解析し、存在しない行番号の折りたたみ状態を破棄することで緩和
- **NSTextView のレイアウト複雑性**: 折りたたみによってテキスト範囲が変わるため、NSRange の計算が複雑になる → `MarkdownStyler` を単一の信頼できるソースとし、折りたたみ後の文字列から Range を計算する
- **パフォーマンス**: 折りたたみ変更のたびに全体を再スタイリングする → ノートは通常小さいため許容範囲
